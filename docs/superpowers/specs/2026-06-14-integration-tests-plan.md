# Интеграционные тесты: полный путь бот → бэкенд → VDS

**Date:** 2026-06-14
**Status:** Plan (к исполнению в свежей сессии)
**Контекст:** ревью рефакторинга reconcile-модели (`2026-06-14-mtprotokey-reconcile-model-design.md`, коммиты `c1a1594..83e8bcb`). Цель — не unit-, а **чёрнобоксовые интеграционные тесты живого стенда**, проверяющие весь путь запроса и его эффект вплоть до VDS-инстанса.

---

## Цель

Покрыть **полный путь** каждого пользовательского сценария:

```
[fake aiogram отсутствует] → bot domain-client (bot/src/domains/*) 
   → РЕАЛЬНЫЙ бэкенд (HTTP, Bot-Auth-Token) 
   → РЕАЛЬНЫЙ Celery (push_key_to_servers_task) 
   → РЕАЛЬНЫЙ VDS-инстанс (my-mtproto-vds-instance, FastAPI + telemt)
```

Проверяем не только HTTP-ответ бэкенда, но и **фактическое состояние на всех VDS** (секрет появился/обновился/удалился).

---

## Принятые решения (владелец, 2026-06-14)

1. **Драйвер = domain-клиенты бота.** Тесты импортируют и зовут реальные клиенты из `bot/src/domains/*` (`FreeTrialClient`, `ReferralsClient`, `LinksClient`, `PaymentsClient`), `BackendClient.base_url` указывает на запущенный бэкенд. Telegram/aiogram-слой и хендлеры **не** задействуются (мокать «сторону бота» = просто не дёргать aiogram). Это покрывает контракт бот↔бэкенд (поймало бы, например, удаление поля `link`) + весь путь до VDS.
2. **Проверка VDS — через GET (готов).** Владелец добавил `GET /api/users/{username}` в `my-mtproto-vds-instance` (`200 {username, link}` / `404`); тесты проверяют присутствие/отсутствие и ротацию секрета через него. См. «Предусловие» ниже — контракт зафиксирован.
3. **Мокать максимум — запросы со стороны бота.** Всё остальное (бэкенд, Celery, VDS) — реальное, на запущенных контейнерах владельца.

---

## ⚠️ Безопасность данных (критично)

Тесты идут по **живому стенду**. В проде ~1500 реальных пользователей — туда тесты пускать НЕЛЬЗЯ.

- Гонять **только против локального/стейдж-стека** (`docker-compose.local.yml`) с изолированной/пустой БД. **VDS-инстанс тоже развёрнут локально** (`../my-mtproto-vds-instance`, свой `docker-compose.local.yaml`) — весь путь бэкенд→VDS реальный и локальный.
- Сетап-условие: в БД бэкенда должны быть `VDSInstance`-строки, чьи `internal_ip_address`/`port` указывают на **локальный** VDS-контейнер (иначе push/GET уйдут «в никуда» либо на прод-VDS). Харнесс при `arrange` создаёт/проверяет такие строки и убеждается, что GET `internal_url` локального VDS отвечает.
- Все тестовые пользователи — синтетические telegram_id из зарезервированного диапазона (напр. префикс `9990000xxx`), чтобы исключить коллизии.
- Жёсткий teardown после каждого теста: удалить тестовые `SystemUser`/`MTPRotoKey` из БД **и** секреты со всех VDS (DELETE `/api/users`).
- Сценарии с глобальными счётчиками (`GLOBAL_KEYS_LIMIT=1000`, `FIRST_MONTH_LIMIT=50`) реалистичны только на чистой БД — фиксируй это в setup.

**Первое, что должен сделать следующий агент:** подтвердить у владельца, что стенд для тестов — НЕ прод, и какой telegram_id-диапазон безопасен.

## Предусловие: GET-эндпоинт на VDS — ДОБАВЛЕН ✅ (итоговый контракт)

Владелец добавил `GET /api/users/{username}` в `my-mtproto-vds-instance` (проверено 2026-06-14):

- **Пользователь есть** → `200 {"username": "<lower>", "link": "tg://proxy?server=<host>&port=443&secret=ee<token><hex(domain)>"}`.
  Ссылка извлекается из ответа telemt (`data.links`) по приоритету режимов `tls → secure → classic`.
- **Пользователя нет** → telemt отдаёт 404 → `@app.exception_handler(httpx.HTTPStatusError)` пробрасывает тот же статус → `404 {"detail": "User not found"}`.
- Эндпоинт без auth (внутренняя сеть) — харнесс ходит по `internal_url` VDS. username приводится к lower (для telegram_id-цифр — no-op).

### Как ассертить состояние VDS

**Присутствие/отсутствие** (основная масса тестов): GET на каждом активном/здоровом VDS → `200` (есть) / `404` (нет).

**Сверка секрета / ротация** (тест перевыпуска): распарсить query-параметр `secret=` из поля `link` и сверить:
- **надёжно (всегда):** `key.token in secret` — сырой 32-hex токен входит подстрокой в `ee<token><hex(domain)>`, не зависит от конфига telemt;
- **строго (бонус):** `secret == key.get_secret_token()` — сойдётся, только если FakeTLS-домен в конфиге telemt == `settings.TLS_DOMAIN` (`beatvault.ru`). В unit-фикстуре VDS-репо домен `petrovich.ru` — это тестовое значение, не прод.

> `server=` в ссылке — телемтовский хост (IP/`::`), **не** `{name}.beatvault.ru`. Полную ссылку с `key.get_proxy_link(server_name=...)` НЕ сверять — только параметр `secret`.

Формат секрета: `ee` + `{raw token}` (32 hex) + `hex(masking_domain)` — структурно совпадает с `MTPRotoKey.get_secret_token()`.

## Инфраструктура харнесса (открытые вопросы — решить с владельцем)

Для arrange/assert нужен доступ к состоянию бэкенда (создать рефералов, выставить `first_month_free_used`, проставить `last_update` для кулдауна, истечь ключ). Через публичный API это не всё достижимо. Варианты (выбрать в начале сессии):

- **(A) Харнесс с доступом к Django ORM** — pytest запускается внутри/рядом с django-контейнером, делит settings+БД. Arrange/assert через ORM, act — через клиенты бота, verify VDS — через GET. **Рекомендуется** (самый гибкий).
- **(B) Только публичный API** — состояние готовится цепочкой реальных запросов. Чище как «чёрный ящик», но часть corner-кейсов (кулдаун, истечение, лимиты) недостижима без time-travel/ORM.

Также подтвердить: где живёт харнесс (новый каталог `integration_tests/` в backend-репо?), запущен ли `celery-worker` (для async-пуша — да, есть в compose), как ждать async-доставку (поллинг VDS GET с таймаутом ~10s).

## Хелперы (написать в харнессе)

- `wait_until(predicate, timeout=10s, interval=0.3s)` — поллинг для async-доставки (Celery).
- `vds_has(server, username) -> bool` — GET VDS, `200`→True / `404`→False.
- `vds_secret(server, username) -> str` — GET VDS, распарсить `secret=` из поля `link` (для сверки ротации).
- `assert_on_all_healthy_vds(username, present=True)` — проверка по всем активным/здоровым VDS.
- `cleanup(username)` — удалить из БД + DELETE со всех VDS.
- `make_test_id()` — синтетический безопасный telegram_id.

## Список тестов (чек-лист, по флоу)

### 1. Бесплатный ключ — `FreeTrialClient.claim` → `POST /users/first-free-link/`
- [ ] Новый пользователь: ответ `{expired_date}` **без `link`**; `expired_date` = now+30д (формат `%d.%m.%y`).
- [ ] После выдачи секрет присутствует на **всех** здоровых VDS (poll GET).
- [ ] Реферал (`invited_from_username` задан): период 14 дней.
- [ ] Исчерпан `FIRST_MONTH_LIMIT` (50 уже использовавших): период 7 дней. *(нужна чистая БД)*
- [ ] Повторная выдача тому же юзеру → `AlreadyUsedFree` (HTTP 400), новый ключ не создан.
- [ ] Инвариант «одна строка на юзера»: после выдачи у юзера ровно 1 `MTPRotoKey`.
- [ ] `GLOBAL_KEYS_LIMIT` достигнут → `KeysLimitReached` (400), запись не создана, пуш не поставлен. *(чистая БД + наполнение)*

### 2. Проверка доступности — `FreeTrialClient.check_availability` → `POST /users/check-first-free-link/`
- [ ] Новый юзер → `MONTH`.
- [ ] По рефералу → `TWO_WEEK`.
- [ ] Лимит исчерпан → `WEEK`.
- [ ] Уже использовал → `NOT_AVAILABLE`.

### 3. Реферальный кабинет — `ReferralsClient.get_cabinet` → `POST /users/referral/cabinet/`
- [ ] Корректные `total_referrals_count` / `active_referrals_count` / `referral_link` / `link_activated_count`.

### 4. Реферальная награда — `ReferralsClient.claim_reward` → `POST /users/referral/link/`
- [ ] 5+ активированных рефералов → ключ на 14 дней, ответ без `link`, секрет на всех VDS.
- [ ] < 5 активных рефералов → `NotEnoughReferrals` (400).
- [ ] Повторный забор после `REFERRAL_LINKS_LIMIT=1` → `AlreadyUsedProgram` (400).

### 5. Перевыпуск — `LinksClient.reissue` → `POST /users/update-link/`
- [ ] Есть активный ключ → новый `token`, **тот же** `expired_date`; ответ без `link`.
- [ ] Новый секрет доставлен на все VDS; старый заменён: `vds_secret()` содержит новый `key.token` и больше не содержит старый (см. сверку секрета в «Предусловии»).
- [ ] Кулдаун: повторный вызов в течение 5 минут → `TooManyRequests` (400). *(нужен ORM/time-control)*
- [ ] Нет пользователя → `KeyDoesNotExist` (400).
- [ ] Нет активного ключа (истёк / `was_deleted` / `is_active=False`) → `KeyDoesNotExist` (400), пуш не поставлен.
- [ ] Инвариант «одна строка»: лишние ключи юзера удаляются при перевыпуске.

### 6. Мои серверы — `LinksClient.get_my_servers` → `POST /users/my-servers/`
- [ ] Возвращает `expired_date` + `servers[]`, по одному на **каждый активный** VDS.
- [ ] `proxy_link` содержит `{vds.name}.beatvault.ru` и секрет из `get_secret_token()` (домен из `TLS_DOMAIN`).
- [ ] Неактивные VDS исключены из списка.
- [ ] Нет активного ключа → `KeyDoesNotExist` (400).

### 7. Оплата — `PaymentsClient.confirm_purchase` → `POST /payments/buy/`
- [ ] Новый юзер без ключа: выдан 30-дневный ключ, секрет на всех VDS, создан `Payment`.
- [ ] Юзер с активным ключом: срок **продлён** на 30 дней, `token` тот же (ключ не пересоздан), `Payment` создан.
- [ ] Неизвестный пользователь → `BadPaymentData` (400).
- [ ] История платежей сохраняется при последующем перевыпуске/выдаче (`Payment.key` = `SET_NULL`, не каскад).
- [ ] `PaymentsClient.get_card_invoice` / `get_stars_invoice` → `GET /payments/` возвращает корректный продукт.

### 8. Reconcile / отказоустойчивость доставки
- [ ] Один VDS помечен `is_healthy=False`: при выдаче секрет уходит только на здоровые; нездоровый пропущен. *(подготовка через ORM)*
- [ ] Идемпотентность: повторная выдача/пуш не падает (VDS отдаёт 409 → бэкенд глотает).
- [ ] Дневное удаление (`remove_user_keys_daily` / `RemoveExpiredKeysDailyService`): истёкший ключ удаляется со всех VDS (GET → 404). *(триггер таска вручную)*
- [ ] (опц., если достижимо) Падение пуша исчерпывает ретраи → VDS помечается `is_healthy=False` + алерт админу. Скорее всего вне scope чёрного ящика — отметить как пропуск, если требует инъекции сбоя.

### 9. Бэкенд-кросс-чек (для каждого создающего/меняющего ключ сценария)
- [ ] HTTP-ответ ≠ единственная проверка: после действия проверять и строку в БД (через ORM, вариант A), и состояние VDS (GET).
